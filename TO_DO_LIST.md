## 4. Главные подтвержденные проблемы

### 4.1. Критический дефект: карточка заказа зацикливает возврат

`Подтверждено`

Симптом:

- пользователь открывает заказ;
- при возврате в список может снова автоматически попасть в ту же карточку;
- это выглядит как "зависание" и невозможность выйти.

Причина:

- выбранный заказ хранится в `sessionStorage`;
- список заказов при монтировании автоматически редиректит в сохраненный `selectedOrderId`;
- возврат назад не очищает это состояние.

Что это значит для backlog:

- это не UX-мелочь, а P0-баг;
- нужно исправлять жизненный цикл `selectedOrderId`, а не только "добавить кнопку назад".

Рекомендуемое решение:

1. Убирать `selectedOrderId` при входе в детальную страницу через полноценный route.
2. Очищать `selectedOrderId` перед `navigate(-1)` и перед переходом в `/workzone/chapan/orders`.
3. Убрать автопереход из списка в карточку при обычном открытии страницы списка.

---

### 4.2. Модель приоритета была спроектирована неверно и сейчас находится в миграционном переходе

`Подтверждено`

Историческая проблема:

- в проекте использовалось единое поле `priority: normal | urgent | vip`;
- в UI слово `VIP` уже фактически заменялось на "Требовательный";
- пользовательский смысл другой: "срочный" и "требовательный" должны существовать одновременно, а не взаимоисключающе.

Что подтвердилось повторной проверкой:

- новая модель уже начала внедряться;
- в коде появились `urgency: normal | urgent` и `isDemandingClient: boolean`;
- legacy `priority` пока сохранён для обратной совместимости и fallback-логики.

Следствие:

- архитектурное направление теперь правильное, но переход ещё не завершён;
- пункт "убрать кнопку требовательный клиент и привязать требовательность к обычным и срочным" уже решается не UI-хаками, а через доменную модель;
- остаточные места со старым `priority` всё ещё создают риск неконсистентного поведения.

Правильная целевая модель:

- `urgency: normal | urgent`
- `isDemandingClient: boolean`

Опционально:

- `attentionNote: string | null`

Тогда:

- `urgent` отвечает только за красный алерт, сортировку и SLA;
- `isDemandingClient` отвечает только за отдельный бейдж, подсказки и повышенное внимание менеджера/цеха.

## 11. Пошаговый implementation plan по спринтам

Ниже не "идеальный agile-ритуал", а практический план внедрения с нормальной технической последовательностью.

Логика порядка такая:

1. Сначала убираем критические баги и риски потери данных.
2. Потом чиним модель данных.
3. Потом строим поверх нее UI, финансы и интеграции.
4. Потом шлифуем UX, mobile и вторичные улучшения.

Принцип:

- не строить Google Sheets, пока заказ не стал полноценной сквозной сущностью;
- не чинить "требовательного клиента" только в интерфейсе, пока не разведены доменные поля;
- не допиливать красивый UI карточек, пока не зафиксирован состав данных, которые они должны показывать.

### Sprint 0. Подготовка и фиксация baseline

Цель:

- зафиксировать текущее состояние, чтобы дальше не ломать систему вслепую.

Задачи:

- описать текущие user flows: `Новый заказ`, `Список заказов`, `Карточка заказа`, `Производство`;
- завести технические задачи по эпикам `A-H`;
- зафиксировать текущую схему заказа, позиции, production task и payment;
- собрать список полей, которые пользователь считает обязательными бизнес-данными;
- выделить 3 среды проверки: desktop manager, production user, mobile.

Артефакты:

- карта flow-ов;
- список API и DB полей;
- чеклист smoke-проверок.

Результат спринта:

- есть единый baseline, от которого можно безопасно рефакторить.

## 13. Риски реализации

### R1. Сломать существующие заказы при миграции модели

Риск:

- старые значения `priority` и старые item-данные могут интерпретироваться неверно.

Снижение риска:

- писать явную миграцию;
- иметь fallback mapping;
- прогонять staging migration на копии данных.

### R2. Усложнить форму заказа без выстраивания модели

Риск:

- UI станет богаче, но данные останутся несогласованными.

Снижение риска:

- не добавлять новые вычисления и поля в production, пока они не поддержаны backend.

### R3. Сделать Google Sheets "для галочки"

Риск:

- появится одноразовая интеграция без надежности, обновлений и защиты от дублей.

Снижение риска:

- строить интеграцию только после нормализации модели заказа;
- делать server-side sync, а не фронтовый экспорт.

### R4. Слишком рано тратить время на косметику

Риск:

- команда уйдет в UI-полировку, пока P0/P1 проблемы не закрыты.

Снижение риска:

- сперва спринты 1-5, потом остальное.
